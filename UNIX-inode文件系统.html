📋 场景一：inode 索引节点——文件系统的档案管理系统
目录区（用户可见）
📄 file1 → inode #1001
📁 docs/ → inode #1005
目录项只存储：
• 文件名（用户看到的名字）
• inode 号（指向 inode 表）
不存储文件内容！
inode 表（文件档案）
inode #1001（file1）
权限：rw-r--r--
大小：4096 bytes
链接计数：1
数据块指针：→ Block[0]
inode 存储：权限、大小、时间戳、链接计数、数据块地址
不存文件名！
数据块区（文件内容）
Block[0] ← inode#1001
"Hello, inode!"
文件实际内容...
数据块存储文件的
实际内容，由 inode
中的指针索引。
寻址流程
用户访问 "file1" → 目录区找到 inode #1001 → inode 中读取数据块地址 → 访问 Block[0] → 读取文件内容 ✅
🔗 场景二：硬链接（Hard Link）——多个名字指向同一档案
目录区
📄 file1 → inode #1001
📄 file_hard → inode #1001 ✓
硬链接：新文件名指向同一个 inode，无新 inode，无新数据块！
inode 表
inode #1001（共享）
权限：rw-r--r--
大小：4096 bytes
链接计数：2 ↑ 增加了！
数据块指针：→ Block[0]
链接计数 = 指向该 inode 的目录项数量
数据块区
Block[0] ← file1 和 file_hard 共用
"Hello, inode!"
两个名字，同一内容
硬链接不会占用
额外数据块空间！
硬链接限制
❌ 不能跨分区（inode 号在同一文件系统内唯一）
❌ 不能指向目录（防止目录树循环）
✅ 仅适用于普通文件
🔀 场景三：软链接（Symbolic Link）——独立的快捷方式
目录区
📄 file1 → inode #1001
🔗 file_soft → inode #1002
软链接是独立文件，有自己的 inode，数据块内容只是目标路径字符串
inode 表
inode #1001（file1）
链接计数：1（软链接不增加）
数据块：→ Block[0]
inode #1002（file_soft，独立）
大小：10 bytes（路径长度）
链接计数：1
数据块：→ Block[99]（存路径）
数据块区
Block[0] ← inode#1001
"Hello, inode!"
原文件内容
Block[99] ← inode#1002（软链接）
"/path/to/file1"
← 仅存路径字符串！
软链接断链：删除 file1 后
❌ file_soft 变成悬挂链接（dangling link），指向的路径不存在，访问失败
→ 这是软链接与硬链接最关键的区别：软链接依赖原文件存在
🗑️ 场景四：删除机制——链接计数归零才真正释放
演示：先有 file1 + file_hard（链接计数=2）
步骤一：删除 file1
📄 file1 → inode #1001 ← 删除
📄 file_hard → inode #1001 ✓
inode#1001 链接计数：2 → 1
数据块仍然保留！
file_hard 可正常访问
步骤二：再删除 file_hard
📄 file_hard ← 删除
inode#1001 链接计数：1 → 0
inode 释放！数据块释放！
文件彻底从磁盘删除 ✓
软链接删除对比
删除软链接（file_soft）：inode#1002 计数归0，仅释放软链接自身的 inode 和小数据块（路径字符串），原文件 file1 完全不受影响
📊 场景五：硬链接 vs 软链接核心对比
对比项硬链接（Hard Link）软链接（Symbolic Link）
独立 inode❌ 共用原 inode✅ 有独立 inode
独立数据块❌ 共用原数据块✅ 有独立数据块（仅存路径）
链接计数增加原 inode 链接计数不影响原 inode 计数
跨文件系统❌ 不能跨分区✅ 可以跨分区
指向目录❌ 不支持✅ 可以
原文件删除后✅ 仍可访问（计数>0）❌ 断链失效
类比多个门牌指向同一房子指路纸条，指向另一地址
💡 核心本质：硬链接指向文件本体（inode），软链接指向文件路径（字符串）
📚 核心知识点
inode 存储什么？
存储文件元数据：权限、大小、时间戳、链接计数、数据块指针。不存储文件名！文件名存在目录项中。
为什么 inode 不存文件名？
支持硬链接（多个文件名指向同一 inode）。实现文件重命名只需修改目录项，不需要移动文件数据。
链接计数（nlink）
记录指向该 inode 的目录项数量。只有 nlink 归零时，文件数据才会被真正释放。软链接不影响原文件的 nlink。
inode 的直接/间接寻址
小文件：直接指针指向数据块。大文件：通过一级、二级、三级间接指针扩展寻址空间，支持超大文件存储。
